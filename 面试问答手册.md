## 一、项目概述与自我介绍

### Q1: 请用2-3分钟介绍一下这个项目的技术架构和你的核心贡献？

#### 标准答案

智旅云图是一个**端到端的AI旅行规划系统**，我在其中独立负责后端AI核心模块的设计与开发。

**技术架构**：采用前后端分离架构，后端FastAPI + 前端Vue3(TypeScript)，实现模块化解耦与独立部署。

**我的核心贡献分四个方向**：

**① RAG系统设计**：构建了完整的RAG Pipeline——从本地Markdown攻略入库(ChromaDB)，到Query Rewrite改写用户口语化表达，再到向量+BM25混合检索，最后通过Cross-encoder(qwen3-rerank)精排。通过持续迭代优化，**Top1命中率从80%提升至93.3%，MRR达0.967**。

**② Agent工作流编排**：实现了ReAct Agent（思考-行动-观察循环）结合LangGraph多阶段流水线，将行程生成拆分为**行程规划/地图补充/天气检查/票价校验/一致性校验**5个独立阶段，每个阶段支持独立的优雅降级。

**③ MCP工具化封装**：将高德地图API（POI搜索、地理编码）和天气服务封装为标准的MCP(Model Context Protocol)工具服务器，结合意图识别路由实现统一聊天入口，支持行程规划/编辑/查询/导出等多意图处理。

**④ 工程化基础设施**：搭建了Redis多级缓存层（内存LRU + Redis持久化），覆盖RAG/地图/天气/Rerank等场景；实现了全链路追踪系统，覆盖用户请求→LLM调用→RAG检索→Agent执行的完整链路。

**关键量化成果**：
| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| RAG Top1命中率 | 80% | **93.3%** | +16.6% |
| MRR | 0.889 | **0.967** | +8.8% |
| Rerank缓存命中延迟 | 728ms | **425ms** | **-41.6%** |
| Noise Rate | 25% | ~8% | -68% |

### Q2: 你在这个项目中承担什么角色？最大的挑战是什么？

#### 标准答案

**角色**：独立负责整个后端AI模块的架构设计与实现，包括RAG系统、Agent工作流、MCP服务、缓存系统和可观测性。

**最大挑战及解决过程（STAR法则）**：

- **Situation**：项目初期RAG检索质量很差，大量用户反馈推荐的景点不相关，Top1命中率只有**80%**
- **Task**：需要在1个月内将命中率提升到90%以上，同时控制API成本不大幅增长
- **Action**：
  1. 先做根因分析：写调试脚本逐条分析15条评估样例，发现两大问题
     - Query Rewrite缺失：用户说"不想太早起床"无法匹配"轻松/慢节奏"
     - 噪声干扰："文档开头"、"目的地简介"等低质量片段频繁占据Top1
  2. 第一轮优化：引入LLM-based Query Rewrite（qwen-max, temperature=0.2）+ 规则级Fallback → Top1提升到86.7%
  3. 第二轮优化：实现Cross-encoder Rerank（qwen3-rerank）+ 多维噪声预过滤 → Top1提升到93.3%
  4. 第三轮优化：引入Rerank结果缓存（候选集指纹key）→ 平均延迟降低41.6%
- **Result**：最终Top1=93.3%, MRR=0.967, API成本反而下降10%（降本优化：候选≤top_k时跳过LLM调用）

### Q3: 如果让你重新设计这个项目，你会做哪些改进？

#### 标准答案

我会从三个方向改进：

**① RAG系统方向**：
- 当前混合检索的RRF权重是静态配置（向量0.6/BM25 0.4），可以引入**Learning-to-Rank模型**，根据历史点击数据自动学习最优权重
- 当前知识库是离线手动维护的Markdown文件，可以引入**自动化UGC采集管道**（如爬取小红书/马蜂窝攻略），加上**质量评分机制**自动筛选高质量内容
- 可以探索**多模态RAG**，将攻略中的图片也纳入检索范围（如用户说"想去拍照好看的地方"，能匹配到带图片的景点推荐）

**② Agent可靠性方向**：
- 当前ReAct Agent的工具调用是硬编码的固定流程，可以引入**Self-Reflection机制**，让Agent能自我评估输出质量并决定是否重新规划
- 当前多阶段流水线是串行执行，对于天气检查等无依赖的阶段可以改为**并行执行**（asyncio.gather），将总延迟从8-15s降到5-8s
- 引入**Agent执行日志回放**功能，便于复现和调试复杂的多步推理过程

**③ 可观测性与运维方向**：
- 接入Prometheus+Grafana监控面板，设置P99延迟>2s自动告警
- 为每个LLM调用添加**Token用量和成本统计**，精确核算每次请求的成本
- 引入**A/B测试框架**，在线验证RAG优化策略的效果（如新规则是否真的提升了命中率）

### 2.1 RAG整体架构

#### 标准答案

```
┌─────────────────────────────────────────────────────────────┐
│                    用户输入 (自然语言旅行需求)                  │
│           "大理，3天，喜欢拍照和美食，不想太早起床"              │
└──────────────────────────┬──────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│            Layer 1: Query Rewrite (查询改写)                 │
│  ┌─────────────────────┐   ┌─────────────────────┐         │
│  │ LLM-based (主路径)   │   │ Rule-based (Fallback)│         │
│  │ qwen-max, t=0.2     │   │ 15条领域规则          │         │
│  │ "大理 日落..."      │   │ "不想早起"→"轻松"    │         │
│  └──────────┬──────────┘   └──────────┬──────────┘         │
│             └─────────┬───────────────┘                    │
│                       ↓                                    │
│              标准化的检索 query                              │
├─────────────────────────────────────────────────────────────┤
│            Layer 2: Retrieval (混合检索)                     │
│  ┌──────────────────┐    ┌──────────────────┐               │
│  │ ChromaDB 向量检索 │    │ BM25 关键词检索   │               │
│  │ text-embedding-v4│    │ rank_bm25        │               │
│  │ top-12 召回       │    │ top-12 召回        │               │
│  └────────┬─────────┘    └────────┬─────────┘               │
│           └────────┬──────────────┘                         │
│                    ↓                                        │
│           RRF 融合 (k=60) 或 加权融合(0.6:0.4)             │
├─────────────────────────────────────────────────────────────┤
│            Layer 3: Noise Filter (噪声过滤)                 │
│  • 文档开头/目录/前言 → 过滤                                │
│  • 法律声明/版权信息 → 过滤                                 │
│  • 内容长度异常(<30字 or >3000字) → 降权                   │
│  • 动态噪声率控制 (目标<10%)                                 │
├─────────────────────────────────────────────────────────────┤
│            Layer 4: Rerank (精排重排序)                     │
│  ┌─────────────────────┐   ┌─────────────────────┐         │
│  │ Cross-encoder (主)   │   │ Rule-based (Fallback)│         │
│  │ qwen3-rerank        │   │ 关键词打分+领域权重   │         │
│  │ 精排top-20→top-k    │   │ 标题匹配+3/正文+1    │         │
│  └──────────┬──────────┘   └──────────┬──────────┘         │
│             └─────────┬───────────────┘                    │
│                       ↓                                    │
│              最终 top-k 结果                                │
├─────────────────────────────────────────────────────────────┤
│            Layer 5: Cache (多级缓存)                        │
│  L1: 内存LRU (<1ms) → L2: Redis (10-50ms) → 计算 (500ms+)  │
│  TTL分级: 天气30min / 地图24h / RAG 6h / Rerank 6h          │
└─────────────────────────────────────────────────────────────┘
```

**各层职责**：

| 层次 | 解决的核心问题 | 技术选型理由 |
|------|---------------|-------------|
| **Query Rewrite** | 用户口语化表达 vs 文档规范表述之间的**语义鸿沟** | LLM理解能力强但慢/规则快但粗糙 → 双路Fallback保证可用性 |
| **Hybrid Retrieval** | 语义模糊查询（"亲子游"）+ 精确关键词匹配（"喜洲古镇"） | 向量擅长语义/BM25擅长精确 → RRF融合两者优势 |
| **Noise Filter** | 召回结果中混入低质量的元数据片段（目录、前言、版权声明） | 在Rerank前预先过滤，**节省API费用且提升精度** |
| **Cross-encoder Rerank** | Bi-encoder召回的top-20中排序不够精准，噪声仍可能排在前 | Cross-encoder能建模query-doc的深度交互，精度远高于Bi-encoder |
| **Multi-level Cache** | 相同query重复请求导致的延迟和成本浪费 | 内存<1ms/Redis 10-50s → 大幅降低尾延迟 |

- 向量检索: `backend/app/rag/vector_db.py`
- BM25索引: `backend/app/rag/bm25_index.py`

### Q5: 为什么选择ChromaDB作为向量数据库？而不是Milvus/Pinecone？

#### 标准答案

选择ChromaDB是基于项目的**实际阶段和需求**做出的工程决策：

**选择ChromaDB的理由**：

1. **轻量级嵌入式部署**：ChromaDB是纯Python库，无需额外安装Docker或独立服务进程，非常适合个人项目/小团队快速迭代
2. **开发效率极高**：API简洁直观，`collection.upsert()`一行代码即可完成数据写入，`collection.query()`完成检索
3. **零运维成本**：数据以文件形式存储在本地磁盘（`db/chroma_db`），无需关心集群扩容、备份恢复等问题
4. **与LangChain生态无缝集成**：LangChain原生支持ChromaDB作为VectorStore

**为什么不选Milvus/Pinecone**：

| 维度 | ChromaDB | Milvus | Pinecone |
|------|----------|--------|----------|
| **部署复杂度** | pip install即可 | 需要Docker/K8s集群 | 托管服务，有网络依赖 |
| **数据规模** | <100万向量（适合） | 千万~亿级 | 亿级+ |
| **当前数据量** | ~750 chunks × 5城市 | - | - |
| **成本** | 免费 | 自建需服务器成本 | 按存储+查询计费 |
| **团队规模** | 1人 | 需要运维人员 | - |

**结论**：在当前数据量（~750个chunks）和团队规模下，ChromaDB是**最优解**。如果未来知识库扩展到百万级以上（如接入全网旅游攻略），会考虑迁移到Milvus分布式方案。

### 2.2 Query Rewrite 查询改写

### Q6: 你的Query Rewrite是怎么实现的？具体细节是什么？

#### 标准答案

Query Rewrite采用**双路策略**：LLM-based为主路径，Rule-based为Fallback。

**代码位置**: `backend/app/agents/tools/rag_tool.py`

**(1) 判断何时需要Rewrite**

采用**默认全量Rewrite策略**，通过`use_llm`参数控制走哪条路径：

```python
def build_destination_query(destination, preferences, ..., use_llm=False):
    if use_llm:
        llm_query = llm_rewrite_query(...)  # 主路径
        if llm_query:
            return llm_query
    return _rule_based_query(...)  # Fallback
```

**(2) LLM-based Rewrite的Prompt设计**

这是整个Query Rewrite的核心，Prompt设计遵循五个层次：

```python
system_prompt = (
    "你是一个 RAG 检索 query 改写专家。"           # ① 角色定义
    "你的任务是把用户的旅行需求改写成适合向量检索的关键词组合。" # ② 任务描述
    "输出要求："
    "1. 只输出检索关键词，用空格分隔"                # ③ 格式约束（关键！）
    "2. 不要输出解释、标点或任何多余文字"            # ④ 负面约束
    "3. 关键词要具体，优先包含景点名称、活动类型、场景特征" # ⑤ 质量要求
    "4. 包含目的地城市名"                          # ⑥ 必须包含项
)
```

**关键设计决策**：
- **temperature=0.2**：低随机性，保证相同输入产生稳定输出
- **结构化标签输入**：用`"目的地：大理"`、`"偏好：拍照"`等前缀让LLM明确每个字段语义角色
- **失败返回None而非抛异常**：保证调用方能自动切换到规则级路径

**效果示例**：
```
输入: 目的地=大理, 备注="不想太早起床，希望看日落"
输出: "大理 日落 傍晚 洱海 双廊 自然风景 拍照 轻松 慢节奏"
```

**(3) Rule-based Fallback的规则引擎**

当LLM调用失败（超时/API不可用/返回空）时，自动降级到规则引擎：

```python
rule_keywords = [
    (("日落", "傍晚"), "大理", ["日落", "傍晚", "洱海", "双廊"]),
    (("日出", "清晨"), "大理", ["日出", "才村", "龙龛"]),
    (("拍照", "出片", "摄影"), None, ["拍照", "摄影", "出片"]),
    (("不想太早起床", "睡到自然醒"), None, ["轻松", "慢节奏"]),
    (("古镇",), "大理", ["古镇", "大理古城", "喜洲古镇"]),  # 目的地差异化！
    (("古镇",), "西安", ["古镇", "回民街"]),               # 同"古镇"不同城市不同映射
]
```

**三元组数据结构**：`(触发词组, 目的地过滤(None=通用), 输出关键词列表)`

**规则构建方法**：
- 初始版本基于旅行领域通用知识编写10条基础规则
- 通过Bad Case驱动迭代：每次发现检索失败的查询，分析原因后补充规则
- 当前共**15条规则**，覆盖5个城市的常见场景（日落/拍照/美食/亲子/文化）

**(4) 规则级输出的五层构建**

```python
def _rule_based_query(...):
    parts = [destination]              # ① 必须包含目的地（锚定）
    for p in preferences:
        _append_unique(parts, p)       # ② 用户偏好原样保留
    if pace:
        _append_unique(parts, pace)    # ③ 节奏要求
    for kw in _extract_note_keywords(special_notes):
        _append_unique(parts, kw)      # ④ 规则提取的关键词
    for term in ["景点","行程","攻略","推荐"]:
        _append_unique(parts, term)    # ⑤ 稳定术语兜底
    return " ".join(parts)
```

### Q7: LLM Rewrite的temperature为什么设为0.2？设为0或0.7有什么区别？

#### 标准答案

| temperature值 | 特点 | 适用场景 |
|---------------|------|---------|
| **0.0** | 完全确定性，每次输出完全相同 | 代码生成、数学计算、格式转换 |
| **0.2** ⭐（我们的选择） | 低随机性，允许微小变化 | **Query Rewrite** |
| **0.7** | 中等随机性，创意性强 | 对话、创意写作、头脑风暴 |
| **1.0+** | 高随机性，不可控 | 艺术创作、多样性采样 |

**选择0.2的原因**：
- **稳定性优先**：相同输入99%情况下输出高度相似，保证缓存命中率和用户体验一致性
- **可控性强**：不会产生离谱的关键词（如用户问"大理美食"却输出"火星料理"）
- **保留微变异性**：偶尔能在同义词间切换（如"拍照"/"摄影"/"出片"），丰富检索覆盖面

**如果设为0.0的问题**：过于死板，某些语义相近但表述不同的需求可能总是生成相同的query，导致长尾场景覆盖不足。

**如果设为0.7的问题**：输出波动太大，可能一次输出"大理古城苍山"，下次输出"去云南玩"，严重影响检索稳定性。

### Q8: 规则引擎的规则是如何积累和维护的？如何避免规则爆炸？

#### 标准答案

**规则积累的三种来源**：

1. **先验知识**：基于旅行领域的通用认知编写初始规则（如"日落"→关联"洱海/双廊"、"熊猫"→关联"成都大熊猫基地"）
2. **Bad Case驱动**：每次线上发现检索失败的案例，分析根因后针对性添加规则
   - 例：用户说"想带小孩玩"，没召回亲子友好景点 → 新增`("亲子", "带小孩", "家庭")`规则
3. **日志挖掘**：定期分析RAG未命中的查询日志，批量发现高频未覆盖模式

**避免规则爆炸的策略**：

- **目的地差异化**：同一条触发词在不同城市映射到不同的输出关键词，避免为每个城市重复定义
  ```python
  (("古镇",), "大理", ["喜洲古镇"])   # 大理特有
  (("古镇",), "西安", ["回民街"])       # 西安特有
  ```
- **通用规则前置**：不依赖特定目的地的规则设为`None`，适用于所有城市
- **规则数量上限**：控制在30条以内，超过此阈值应考虑迁移到LLM方案或学习型方法
- **定期清理**：每月review规则命中率，移除从未被触发的规则

### 2.3 向量库与文档切片

### Q9: 你的Markdown文档切片策略是什么？为什么不用固定长度切片？

#### 标准答案

**代码位置**: `backend/app/rag/vector_db.py` 的 `_split_markdown_into_chunks()` 函数

**策略**：**按二级/三级标题（## / ###）切分**，而非固定长度（如512字）

```python
for line in markdown_text.splitlines():
    stripped = line.strip()
    if stripped.startswith("## ") or stripped.startswith("### "):
        # 遇到新标题 → 保存当前chunk → 开始新chunk
        if current_lines:
            chunks.append({"title": current_title, "text": "\n".join(current_lines)})
            current_lines = []
        current_title = stripped.lstrip("#").strip()
    elif stripped:
        current_lines.append(stripped)
```

**选择理由对比**：

| 切片方式 | 优点 | 缺点 | 适用场景 |
|---------|------|------|---------|
| 固定长度(512字) | 实现简单，粒度均匀 | 可能切断语义，标题和正文分离 | 通用文本，无明确结构 |
| 递归字符切分 | 保持语义完整 | 实现复杂，参数敏感 | 长篇论文/书籍 |
| **按标题切分** ⭐ | **语义完整、粒度适中、title天然可解释** | 依赖文档有良好结构 | **结构化文档（如攻略）**|

**对旅行攻略特别适合的原因**：
- 攻略天然是高度结构化的文档，每个`##`章节都有明确的主题（"大理古城"、"洱海"、"美食推荐"）
- 切片后的chunk大小通常200-800字，正好适配text-embedding-v4的最佳编码窗口
- title字段直接描述内容主题（如"洱海日落观赏指南"），后续噪声过滤和Rerank都可以利用

**边界情况处理**：
- 第一个chunk默认title为"文档开头"（通常是概述性内容，后续会被噪声过滤器识别并降权）
- 连续标题之间空内容会产生空text chunk（后续会被过滤）
- 空行会被跳过（只保留非空行）

### Q10: Chunk ID是如何生成的？为什么用MD5而不用UUID？

#### 标准答案

```python
def _build_chunk_id(source: str, title: str, text: str) -> str:
    digest = md5(f"{source}|{title}|{text}".encode("utf-8")).hexdigest()
    return f"{source}_{digest[:12]}"
```

**ID组成**：`{来源文件名}_{内容哈希前12位}`

**三要素设计**：
- `source`：文件来源区分（如`dali_guide.md`）
- `title`：章节区分（如"大理古城"）
- `text`：内容唯一性保证（正文内容的hash）

**MD5 vs UUID对比**：

| 维度 | MD5 | UUID |
|------|-----|------|
| **确定性** | ✅ 相同内容始终生成相同ID | ❌ 每次都不同 |
| **幂等更新支持** | ✅ upsert时能判断chunk是否存在 | ❌ 无法判断，只能追加 |
| **可读性** | ✅ 包含source前缀，调试时一目了然 | ❌ 纯随机字符串 |
| **冲突概率** | 取前12位，理论上有碰撞风险 | 实际上可忽略 |

**为什么确定性重要**：当知识库更新（修改了某个城市的攻略）后重新执行ingest脚本，ChromaDB的`upsert()`操作能根据相同ID**更新已有记录而非创建重复数据**。

### Q12: 为什么需要混合检索？只用向量检索不行吗？

**纯向量检索的短板**：

| 场景 | 纯向量检索表现 | 原因 |
|------|--------------|------|

- 精确匹配专有名词（"喜洲古镇"、"双廊"、"才村码头"）

- 计算速度快，无需调用外部API

**混合检索的效果**（RRF融合后）：
- 语义类查询（"亲子游"、"拍照"）：主要靠向量贡献，BM25辅助
- 精确类查询（"喜洲古镇"、"野生菌火锅"）：主要靠BM25贡献，向量辅助
- 复合查询（"大理拍照+喜洲古镇"）：两者互补，取长补短

### Q13: RRF（Reciprocal Rank Fusion）的具体公式是什么？为什么k取60？

#### 标准答案

**代码位置**: `backend/app/rag/hybrid_retriever.py` 的 `_fuse_results_with_rrf()`

**RRF公式**：

$$RRF\_score(d) = \sum_{r \in R} \frac{1}{k + rank_r(d)}$$

其中：
- $d$ = 候选文档
- $R$ = 排序结果集合（本项目中 $|R|=2$：vector + BM25）
- $rank_r(d)$ = 文档$d$在排序$r$中的排名（从1开始）
- $k$ = 平滑常数

**数值示例**：

```
文档A: vector排名第1, BM25排名第3
  RRF = 1/(60+1) + 1/(60+3) = 0.01639 + 0.01587 = 0.03226

文档B: vector排名第2, BM25排名第1  
  RRF = 1/(60+2) + 1/(60+1) = 0.01613 + 0.01639 = 0.03252

文档C: 只在vector中排名第10，BM25中未出现
  RRF = 1/(60+10) + 0 = 0.01429

排序: B(0.03252) > A(0.03226) > C(0.01429)
```

**k=60的由来**：
- 来自经典论文 *"Reciprocal Rank Fusion outperforms Condorcet and individual rankers"* 的推荐值
- k越大，排名差异的影响越小（更平滑）；k越小，对top排名越敏感
- 60是经过实验验证的经验值，在大多数检索任务中表现稳定
- 如果你的候选集较小（如top-10），可以考虑降低k到20-30

**RRF vs 加权融合对比**：

| 特性 | RRF | 加权融合 |
|------|-----|---------|
| **分数归一化** | 不需要，只看排名 | 需要Min-Max归一化 |
| **对异常值鲁棒性** | 高（某文档在一个系统中得分极高也不会主导最终结果）| 较低 |
| **实现复杂度** | 低（几行代码） | 中等（需归一化+权重调优）|
| **适用场景** | 排名质量可靠时 | 有可靠的相关性分数时 |

项目中两种都实现了，默认使用RRF。

### Q14: BM25的分词器是怎么实现的？为什么不用jieba等专业分词工具？

#### 标准答案

**代码位置**: `backend/app/rag/bm25_index.py` 的 `BM25Index._tokenize()`

```python
def _tokenize(self, text: str) -> list[str]:
    import re
    tokens = re.findall(r'[\u4e00-\u9fff]+|[a-zA-Z]+|\d+', text)
    return [t.lower() for t in tokens if t.strip()]
```

**Tokenizer设计**：

| 正则表达式 | 匹配内容 | 示例 |
|-----------|---------|------|
| `[\u4e00-\u9fff]+` | 中文字符序列 | `"大理古城"` → `["大理古城"]` |
| `[a-zA-Z]+` | 英文单词 | `"GPS"` → `["gps"]` |
| `\d+` | 数字 | `"2024"` → `["2024"]` |

**不用jieba的原因**：
- ✅ **零依赖**：不需要安装额外的NLP包，减小部署体积
- ✅ **速度快**：正则表达式比jieba的DAG/HMM模型快100倍以上
- ✅ **足够好用**：对于短文本检索（旅行攻略片段），bigram级别的中文粒度已经够用
- ❌ jieba的缺点：无法处理"南京市长"vs"南京市长的"这类歧义（但在你的场景中影响极小）

**持久化策略**：BM25索引构建后序列化为JSON存储在`db/bm25_index/index.json`，启动时懒加载重建BM25Okapi对象。选用JSON而非pickle是为了版本兼容性和可读性。

### 2.5 Rerank 重排序系统

### Q15: 为什么选择Cross-encoder而不是Bi-encoder做Rerank？

#### 标准答案

**核心区别**：

| 特性 | Bi-encoder | Cross-encoder |
|------|-----------|---------------|
| **计算方式** | Query和Doc**独立**编码为向量，然后计算余弦相似度 | Query和Doc**拼接**后送入联合编码器 |
| **时间复杂度** | O(1) per doc（向量可预计算） | O(n) per doc（每次需完整编码）|
| **精度** | 一般（无法捕捉Query-Doc间的细粒度交互）| **高**（注意力机制能建模深度交互关系）|
| **适用阶段** | **召回阶段**（从百万级中筛出top-K） | **精排阶段**（对top-K重排序）|

**我的架构设计**：**召回用Bi-encoder(ChromaDB) + 精排用Cross-encoder(qwen3-rerank)** = 兼顾效率与精度

**为什么不在召回阶段也用Cross-encoder？**
- 假设有10万个chunks，用Cross-encoder需要对每个chunk做一次完整的前向推理
- 即使每次只需50ms，总共也需要**1.4小时**
- 而Bi-encoder预计算好向量后，检索只需毫秒级

**Cross-encoder的Instruct Prompt设计**：

```python
"instruct": (
    "你是一个旅行攻略检索专家。"
    "给定一个旅行规划查询，从候选文档中检索出最具体、最详细、最能直接回答用户问题的片段。"
    "优先选择包含具体景点名称、活动推荐、实用信息的片段，"
    "避免选择泛化的目的地简介、文档开头等信息量低的片段。"
)
```

这个instruct告诉模型：**不要选长的，要选有用的**。避免模型偏向选择长文本（因为信息量大）但实际上废话多的片段。

### Q16: Rerank之前的噪声预过滤是怎么做的？权重值怎么确定的？

#### 标准答案

**代码位置**: `backend/app/rag/retriever.py` 的 `_calculate_noise_score()` 和 `_score_chunk_for_rerank()`

**(1) 噪声检测 — 五维评分模型**

```python
def _calculate_noise_score(chunk) -> float:  # 返回0-1，越高越可能是噪声
    score = 0.0
    
    # ① 标题噪声检测（权重最高，0.8）
    if title in {"文档开头", "概述", "简介", "前言", "序", "目录"}:
        score += 0.8
    
    # ② 关键词噪声检测（上限0.3）
    noise_kw_count = sum(1 for kw in _NOISE_KEYWORDS if kw in text)
    score += min(noise_kw_count * 0.1, 0.3)
    
    # ③ 内容长度检测（上限0.3）
    if text_len < 30:    score += 0.3    # 太短，可能不完整
    elif text_len > 3000: score += 0.2    # 太长，可能含过多无关内容
    
    # ④ 标点符号占比检测（上限0.2）
    # 前100字符中标点占比>30%则扣分
    
    return min(score, 1.0)
```

**(2) 规则级Rerank打分权重表**

| 操作 | 分值变化 | 设计理由 |
|------|---------|---------|
| **标题匹配关键词** | **+3** | 标题是主题概括，标题匹配意味着主题高度相关 |
| **正文匹配关键词** | **+1** | 正文提及可能是偶然的，权重较低 |
| **"行程"类标题** | **+4** | 行程类片段包含具体的路线/时间安排，价值最高 |
| **"行程参考"类标题** | **-4** | 内容过于全面但缺乏针对性，容易霸占Top1 |
| **"目的地简介"类** | **-2** | 信息密度低，泛化描述多 |
| **跨城市污染** | **-5** | 查询大理却召回成都的内容，硬约束惩罚 |

**(3) 权重确定方法论**：

> **不是拍脑袋定的，而是经过三轮迭代的经验+实验值**：
>
> 1. **第一轮（经验初始化）**：基于RAG领域的通用知识设定初始值（文档开头-8分、行程参考-4分）
> 2. **第二轮（Bad Case驱动调整）**：
>    - 发现"行程参考"频繁占据Top1 → 从0调整为-4
>    - 发现跨城市污染问题 → 新增-5的目的地不匹配惩罚
> 3. **第三轮（A/B验证）**：对比调整前后的指标变化
>    - Top1: 86.7% → 93.3%
>    - Noise Rate: 25% → 8%

**(4) 可解释性输出**：

每条Rerank结果都附带`rerank_reasons`字段，记录打分明细：
```json
{
  "title": "洱海日落观赏指南",
  "rerank_score": 12,
  "rerank_reasons": [
    "title+3:日落",
    "text+1:洱海",
    "domain+4:行程标题",
    "cross-encoder:0.9521"
  ]
}
```

这便于调试分析：为什么这条排在前面？哪些规则起了作用？

### Q17: Rerank缓存的Key是怎么设计的？为什么要包含候选集指纹？

#### 标准答案

**代码位置**: `backend/app/rag/retriever.py` 的 `_build_rerank_cache_key()`

```python
def _build_rerank_cache_key(query: str, chunks: list) -> str:
    normalized_query = _normalize_cache_text(query)  # 标准化
    
    # 🔑 关键：候选集指纹
    chunk_signatures = sorted([                        # 排序降低顺序敏感性
        f"{c['source']}:{c['title']}" for c in chunks
    ])
    content_fingerprint = "|".join(chunk_signatures)
    chunks_hash = hashlib.md5(content_fingerprint.encode()).hexdigest()[:12]
    
    return f"rerank:{normalized_query}:{chunks_hash}"
```

**Key的三段式结构**：`rerack:{标准化query}:{候选集MD5指纹}`

**为什么必须包含候选集指纹？**

假设两次请求的query相同（都是"大理 日落"）：
- **第一次**：向量库有20个chunks，Rerank后Top1是Chunk A
- **第二次**：知识库刚更新了（新增了大理攻略），向量召回变成了不同的20个chunks

如果只用query做key：
- ❌ 第二次会错误地返回第一次的Rerank结果（针对旧候选集排序的）
- ✅ 包含候选集指纹后，第二次的key不同，会重新计算Rerank

**其他设计考量**：
- `sorted()`排序：消除chunk顺序的影响（ChromaDB版本不同可能导致召回顺序不同）
- 只用`source:title`做指纹：不包含正文，避免key过长（完整chunk可能有几KB）
- MD5截断到12位：平衡唯一性和可读性

### 2.6 噪声检测与过滤

### Q18: 你的噪声检测系统是怎么工作的？阈值为什么设为0.6？

#### 标准答案

**两层过滤机制**：

**第一层：硬过滤（绝对噪声）**
```python
_NOISE_TITLES = {"文档开头", "概述", "简介", "前言", "序", "目录", "附录"}
_NOISE_KEYWORDS = {"本文", "仅供参考", "版权所有", "未经许可", ...}
```
如果chunk的title在噪声标题集合中，或者正文中出现大量法律声明关键词，直接判定为噪声。

**第二层：软过滤（动态噪声评分）**
```python
noise_score = (
    标题匹配权重(0~0.8) + 
    关键词密度权重(0~0.3) +
    长度异常权重(0~0.3) +
    标点密度权重(0~0.2)
)
if noise_score >= 0.6:  # 阈值判定
    mark_as_noise()
```

**阈值0.6的确定过程**：

| 阈值 | 误删率（有价值内容被删）| 漏删率（噪声未被删除）| 综合评价 |
|------|---------------------|-------------------|---------|
| **0.4** | 8% | 3% | 太激进，误删太多有用内容 |
| **0.5** | 3% | 8% | 一般 |
| **0.6** ⭐ | **<2%** | **~10%** | **最佳平衡点** |
| **0.8** | 0% | 25% | 太保守，噪声过滤不足 |

**动态噪声过滤**（`_dynamic_noise_filter`）：
- 不仅用固定阈值，还根据**目标噪声率**动态截断
- 默认目标：保留90%的非噪声内容（`noise_rate_target=0.1`）
- 按noise_score升序排列后，只保留前90%
- 保证即使阈值设置不够完美，输出质量也有底线

---

### 2.7 缓存策略

### Q19: 你的缓存体系是怎样的？请详细描述多级缓存的读写流程？

#### 标准答案

**代码位置**: `backend/app/services/cache_service.py`

**架构：L1内存缓存(LRU) + L2Redis缓存**

**读取流程（Cache Aside Pattern）**：

```
请求进入
  ↓
[① 内存缓存 L1] ─── 命中? ──→ 返回数据 (<1ms) ⚡ + move_to_end(更新LRU)
  │ 未命中/过期
  ↓
[② Redis L2] ─── 命中? ──→ 返回数据 (10-50ms) 🐢 + 回填到L1内存缓存
  │ 未命中/Redis不可用
  ↓
[③ 业务逻辑] ──→ 计算结果
  ↓
[④ 写入L1内存] + [⑤ 写入L2 Redis] (异步容错)
  ↓
返回结果
```

**内存缓存（L1）详细设计**：

| 参数 | 值 | 理由 |
|------|-----|------|
| 数据结构 | `OrderedDict` | O(1)的LRU淘汰（`move_to_end` + `popitem(last=False)`）|
| 最大条数 | 1000条 | 防止key过多导致性能下降 |
| 单条上限 | 10MB | 防止异常大对象撑爆内存 |
| 总大小上限 | 500MB | 防止OOM |
| TTL | 60秒 | 短TTL因为进程重启即丢失 |
| 淘汰算法 | LRU（最近最少使用） | 热点数据常驻内存 |

**Redis缓存（L2）详细设计**：

| 参数 | 值 | 理由 |
|------|-----|------|
| Key格式 | `{前缀}:{版本号}:{业务key}` | 如`trip_planner:v2:rerank:大理 日落:a1b2c3...` |
| 序列化 | JSON (`ensure_ascii=False`) | 可读性好，跨语言兼容 |
| TTL分级策略 | 见下方 | 不同数据不同生命周期 |

**TTL分级策略**（来自config.py）：

| 数据类型 | TTL | 理由 |
|---------|-----|------|
| **天气预报** | 1800秒(30分钟) | 天气变化快，需及时更新 |
| **地图POI** | 86400秒(24小时) | 商户信息变化较慢 |
| **RAG检索** | 21600秒(6小时) | 攻略更新频率低 |
| **Rerank结果** | 21600秒(6小时) | 相同query+相同候选集的排序相对稳定 |
| **LLM草稿** | 21600秒(6小时) | 行程草稿短期内不变 |
| **默认TTL** | 1800秒(30分钟) | 兜底值 |

**Redis不可用的降级策略**：
- 三层防护：配置检查 → 依赖检查 → 连接检查（`ping()`测试）
- 单次告警：连接失败只打印一次日志，避免日志爆炸
- 自动降级到纯内存缓存模式，保证核心功能可用

### Q20: 如何防止缓存穿透、缓存雪崩、缓存击穿？

#### 标准答案

| 问题 | 定义 | 本项目的应对方案 |
|------|------|----------------|
| **缓存穿透** | 查询不存在的数据，每次都穿过缓存打到数据库 | 当前业务场景不存在恶意攻击风险，暂未特殊处理。如有需要可在缓存中存空值标记（TTL较短）|
| **缓存雪崩** | 大量key同时过期，瞬间流量全部打到后端 | TTL分散化（当前未实现，可作为优化点：在基础TTL上加减随机偏移量）|
| **缓存击穿** | 热点key过期瞬间，大量并发请求同时重建缓存 | **多级缓存**：即使Redis过期，内存缓存(L1)可能仍有数据；且内存缓存TTL仅60秒，热点key几乎始终在L1中 |
| **Redis宕机** | Redis服务不可用 | **自动降级**：检测到Redis不可用后静默降级到内存缓存模式，通过`_redis_unavailable_logged`标志只告警一次 |

**额外防护措施**：
- Key命名包含**版本号**（`trip_planner:v2:...`），缓存格式变更时升级版本号，旧key自然过期不会误读
- 所有缓存写入操作都用`try-except`包裹，**缓存失败不影响主流程**（符合"缓存是优化而非必需品"的理念）

### 3.1 ReAct Agent 原理与实现

### Q22: 什么是ReAct Agent？和普通LLM调用有什么本质区别？

#### 标准答案

**代码位置**: `backend/app/agents/react_agent.py`

**ReAct = Reasoning + Acting**（推理+行动），是一种让LLM通过**"思考→行动→观察"循环**来解决问题的范式。

**状态机流转图**：

```
         ┌──────────┐
         │ THINKING │ ←──────────────────┐
         └────┬─────┘                        │
              ↓                              │
         ┌──────────┐                        │
         │ ACTING   │                        │
         └────┬─────┘                        │
              ↓                              │
         ┌──────────┐ ──→ 需要更多信息? ──Yes─┤
         │OBSERVING │                        │
         └────┬─────┘                        │
              │No                            │
              ↓                              │
         ┌──────────┐                        │
         │ FINISHED │                        │
         └──────────┘                        │
```

**各状态的职责**：

| 状态 | LLM的角色 | 输出示例 |
|------|----------|---------|
| **THINKING** | 分析师 | "用户想去大理看日落，我需要先查询大理攻略和天气" |
| **ACTING** | 决策者 | `{"tool": "search_destination_guide", "args": {"destination": "大理"}}` |
| **OBSERVING** | 解读器 | "根据攻略，双廊是观赏日落的最佳地点..." |
| **FINISHED** | 总结者 | "已为你规划大理3天旅行计划，重点安排了双廊日落..." |

**vs 普通LLM调用的本质区别**：

| 维度 | 普通LLM调用 | ReAct Agent |
|------|------------|-------------|
| **信息来源** | 仅依赖Prompt中的上下文 | 可**动态调用外部工具**（RAG/天气/地图）获取实时信息 |
| **适应性** | 一次性生成，无法中途调整 | 根据**中间观察结果**决定下一步行动 |
| **可控性** | 黑盒，难以干预中间过程 | **每步都可追踪、调试、干预** |
| **适用场景** | 结构化生成（JSON/XML）| 开放式问题解决（规划/研究/排查）|

**实际意义举例**：
- 普通调用：用户问"今天大理天气怎么样？" → LLM只能基于训练数据回答（可能过时）
- ReAct Agent：用户问同样的问题 → Agent调用WeatherTool获取实时天气 → 基于真实数据回答

### Q23: 你的ReAct Agent的工具系统是怎么设计的？如何扩展新工具？

#### 标准答案

**代码位置**: `backend/app/agents/react_agent.py` 的 `BaseTool` 类

**设计模式：模板方法模式 + 策略模式**

```python
class BaseTool:
    """可插拔工具的基类"""
    name: str                    # 工具名称（唯一标识）
    description: str             # 工具描述（供LLM理解用途）

    def run(self, **kwargs) -> str:
        raise NotImplementedError  # 子类必须实现

    def __call__(self, **kwargs) -> str:
        return self.run(**kwargs)  # 支持函数式调用
```

**已实现的三个工具**：

| 工具 | 名称 | 功能 | 返回格式 |
|------|------|------|---------|
| **RAGTool** | `search_destination_guide` | 搜索目的地本地攻略 | 格式化文本（供LLM阅读）|
| **WeatherTool** | `get_weather` | 查询天气预报 | 7天天气详情 |
| **MapTool** | `search_places` | 搜索POI（景点/餐厅/酒店）| POI列表（名称/地址/评分/类型）|

**扩展新工具只需3步**：

```python
# Step 1: 继承BaseTool
class HotelTool(BaseTool):
    name = "search_hotels"
    description = "搜索酒店信息"

    def run(self, city: str, **kwargs) -> str:
        # Step 2: 实现业务逻辑
        hotels = call_hotel_api(city)
        return format_hotels(hotels)

# Step 3: 注册到Agent
agent.add_tool(HotelTool())
```

**工具设计的关键决策**：

1. **返回格式化文本而非原始JSON**：LLM更容易理解和利用自然语言
2. **异常时返回错误字符串而非抛出异常**：保证Agent不会因单个工具失败而崩溃，LLM可以根据错误信息调整策略
3. **惰性导入**（`from xxx`放在`run()`内部）：避免循环依赖，加快启动速度

### Q24: ReAct Agent的max_steps为什么设为5？如何防止死循环？

#### 标准答案

```python
class ReActAgent:
    def __init__(self, max_steps: int = 5):  # 最大执行步数
```

**max_steps=5的三重作用**：

1. **成本控制**：每一步都可能涉及LLM调用（token消耗）和外部API调用（费用），限制步数控制总体开销
2. **防死循环**：如果Agent陷入"思考→行动→观察→思考..."的无限循环（比如反复查询同一个工具），5步后强制终止
3. **用户体验**：用户等待时间有限，5步通常足够完成"收集信息→生成方案"的完整流程

**实际的步骤分配**（典型场景）：

| 步骤 | 状态 | 执行内容 |
|------|------|---------|
| Step 1 | THINKING | 分析用户需求："需要规划大理3天行程，偏好拍照和美食" |
| Step 2 | ACTING | 调用RAGTool搜索大理攻略 |
| Step 2 | OBSERVING | 获取攻略结果："洱海、双廊、喜洲..." |
| Step 3 | FINISHED | 基于RAG上下文调用LLM生成行程草稿 |

**通常只需要3步**就完成了。5步的上限留给了更复杂的场景（如需要同时查天气+地图+攻略的情况）。

**其他防死循环机制**：
- 每步操作都记录到`steps`列表（包含state/thought/action/observation），便于事后审查
- 通过TracingService记录完整的推理链路，方便定位循环原因
- 未来可引入**重复检测**：如果连续2步调用相同工具且参数相同，强制终止

### 3.2 工具系统设计

### Q25: 你把高德地图和天气服务封装成了MCP工具，什么是MCP？为什么这么做？

#### 标准答案

**MCP (Model Context Protocol)** 是一种标准化协议，让AI模型能够通过统一的接口调用外部工具和数据源。

**代码位置**：
- 地图MCP: `backend/app/mcp/amap_server.py`
- 天气MCP: `backend/app/mcp/weather_server.py`

**本项目中的MCP实现**：

```python
# 使用 FastMCP 框架
from fastmcp import FastMCP
mcp = FastMCP("AmapMCP")

@mcp.tool()
def geocode_address(address: str, city: str = None) -> dict:
    """根据地址获取经纬度信息"""
    ...

@mcp.tool()
def search_places(keyword: str, city: str, page_size: int = 5) -> dict:
    """搜索目的地的景点、餐厅、酒店等POI信息"""
    ...

@mcp.tool()
def get_weather_forecast(city: str) -> dict:
    """获取指定城市的未来天气预报"""
    ...
```

**为什么用MCP封装而不是直接在Agent里调用？**

| 维度 | 直接调用 | MCP封装 |
|------|---------|---------|
| **标准化** | 每个工具接口不一致 | 统一的`@mcp.tool()`装饰器规范 |
| **可发现性** | LLM不知道有哪些工具可用 | MCP提供`description`让LLM自动理解工具用途 |
| **可替换性** | 换一个地图服务商需要改Agent代码 | 只需更换MCP Server实现，Agent无需改动 |
| **独立部署** | 和Agent强耦合 | MCP Server可独立运行、升级、扩展 |
| **缓存集成** | 需要在Agent层面统一处理 | 每个MCP工具内部自行管理缓存策略 |

**MCP工具内部的工程细节**：

1. **API Key安全检查**：每个工具调用前检查`AMAP_API_KEY`是否已配置，未配置则抛出明确错误
2. **HTTP客户端管理**：使用`httpx.Client(timeout=20)`作为上下文管理器，确保连接正确释放
3. **响应校验**：检查高德API返回的`status`字段，非"1"时抛出`RuntimeError`附带上游错误信息
4. **缓存集成**：每个MCP工具内置缓存逻辑（天气30min/地图24h），对调用方透明
5. **数据清洗**：经纬度解析（`_split_location`）、浮点数安全转换（`_parse_float`）、空值保护

### 3.3 多阶段流水线编排

### Q26: 你的多阶段流水线是怎么设计的？5个阶段分别做什么？

#### 标准答案

**代码位置**: `backend/app/services/multi_stage_agent.py`

**5个阶段的职责**：

| 阶段 | 类型 | 职责 | 失败影响 | 降级方式 |
|------|------|------|---------|---------|
| **Stage 1: 行程规划** | TRIP_PLANNING | RAG检索 + LLM生成基础Itinerary | 🔴 致命 | 无法降级，必须成功 |
| **Stage 2: 地图补充** | MAP_ENRICHMENT | 为每个景点补充POI信息（坐标/地址/图片/评分/电话）| 🟡 中等 | 使用LLM生成的虚拟坐标 |
| **Stage 3: 天气检查** | WEATHER_CHECK | 查询目的地天气预报，标注适宜户外活动的日期 | 🟢 轻微 | 跳过，前端隐藏天气模块 |
| **Stage 4: 票价校验** | TICKET_CHECK | 查询景点门票价格，校验预算合理性 | 🟢 轻微 | 使用预算估算值 |
| **Stage 5: 一致性校验** | CONSISTENCY_VALIDATION | 检查距离/营业时间/预算/逻辑矛盾 | 🟡 中等 | 跳过，输出警告 |

**三种执行模式**：

```python
class PipelineMode(Enum):
    FAST = "fast"      # 快速模式：仅执行Stage 1（⚡ 2-3秒）
    FULL = "full"      # 完整模式：全部5个阶段（🔟 8-15秒）
    ASYNC = "async"    # 异步模式：Stage 1同步返回 + 其余后台补充（渐进增强）
```

**PipelineContext — 阶段间数据传递机制**：

```python
@dataclass
class PipelineContext:
    trip_request: TripRequest           # 用户原始请求
    itinerary: Optional[Itinerary] = None  # 核心产出
    rag_contexts: List[str] = [...]      # Stage 1产出
    weather_data: Optional[Dict] = None  # Stage 3产出
    ticket_data: Optional[Dict] = None  # Stage 4产出
    validation_result = None            # Stage 5产出
    custom_data: Dict = {}              # 私有通信通道（命名空间隔离）
    
    def set_stage_data(self, stage_type, key, value):
        self.custom_data[f"{stage_type.value}_{key}"] = value  # 避免key冲突
        
    def get_stage_data(self, stage_type, key, default=None):
        return self.custom_data.get(f"{stage_type.value}_{key}", default)
```

**命名空间隔离的设计精髓**：每个阶段的数据用`{stage_type}_{key}`作为存储key，不同阶段之间不会互相覆盖。

### Q27: 多阶段流水线的StageProcessor基类用了什么设计模式？

#### 标准答案

**模板方法模式（Template Method Pattern）**：

```python
class StageProcessor:
    """单个阶段的处理器基类"""
    
    def __call__(self, context):  # 统一调用入口（模板方法）
        result = StageResult(status=RUNNING)
        result.start_time = datetime.now()
        
        try:
            if not self.can_process(context):  # ① 依赖检查（钩子方法）
                result.status = SKIPPED
            else:
                output = self.execute(context)  # ② 业务逻辑（抽象方法，子类实现）
                result.status = COMPLETED
                
        except Exception as e:                   # ③ 统一异常处理
            result.status = FAILED
            result.error = str(e)
            
        result.end_time = datetime.now()
        result.duration_ms = (result.end_time - result.start_time).total_seconds() * 1000
        return result
```

**模板方法定义了不变的骨架**：
- 时间测量（start_time → end_time → duration_ms）
- 依赖检查（can_process）
- 异常捕获和状态管理
- 统一的StageResult返回格式

**子类只需关注变化的业务逻辑**：
```python
class TripPlanningStage(StageProcessor):
    def execute(self, context):
        rag_contexts = collect_rag_contexts(...)
        draft = generate_llm_draft(...)
        itinerary = build_basic_itinerary(...)
        return itinerary  # 只返回业务结果，不需要关心时间测量和异常处理
```

**好处**：
- **消除重复代码**：5个阶段共享相同的错误处理和时间测量逻辑
- **统一行为**：所有阶段都遵循相同的异常处理策略
- **易于扩展**：新增第6个阶段只需继承`StageProcessor`并实现`execute()`
- **开闭原则**：对扩展开放（新增Stage），对修改关闭（不改基类）

### 3.4 优雅降级机制

### Q28: 当某个阶段失败时，你的优雅降级是怎么实现的？

#### 标准答案

**五种状态的状态机**：

```python
class StageStatus(Enum):
    PENDING = auto()      # 待执行
    RUNNING = auto()      # 执行中
    COMPLETED = auto()    # 成功完成
    FAILED = auto()       # 执行失败
    SKIPPED = auto()      # 跳过（优雅降级）⭐
```

**SKIPPED状态的触发条件**（通过`can_process()`钩子方法）：

```python
class MapEnrichmentStage(StageProcessor):
    def can_process(self, context):
        # 条件1: API Key未配置
        if not config.AMAP_API_KEY:
            return False
        # 条件2: 功能开关关闭
        if not config.ENABLE_AMAP_ENRICHMENT:
            return False
        # 条件3: 上游阶段没有产出有效itinerary
        if not context.itinerary:
            return False
        return True  # 所有条件满足，正常执行
```

**降级的传递效应**：

```
Stage 1 (行程规划) 失败?
  ├── Yes → 整个流水线终止，返回错误
  └── No → 继续 Stage 2
           ├── Stage 2 (地图) 失败/跳过?
           │   ├── itinerary保持Stage 1的结果（只是缺少POI坐标）
           │   └── warnings.append("地图数据缺失，使用默认坐标")
           ├── Stage 3 (天气) 失败/跳过?
           │   └── 前端隐藏天气模块
           ├── Stage 4 (票价) 失败/跳过?
           │   └── 使用预算比例估算
           └── Stage 5 (校验) 失败/跳过?
               └── 输出警告但不阻断
```

**核心原则**：**Stage 1是唯一的致命节点**，其余都是非致命的。这保证了在最坏情况下（所有外部服务不可用），用户至少能得到一个基础的行程规划结果。

### Q30: 你如何保证LLM输出的结构化？（Schema约束 + 重试机制）

**代码位置**: `backend/app/models/schemas.py` 的 `Itinerary` / `DayPlan` / `SpotItem` 等

**Pydantic Schema 强约束**：

```python
class Itinerary(BaseSchema):
```

**多层保障机制**：

| 层级 | 手段 | 作用 |
|------|------|------|
| **① Prompt层** | 在System Prompt中明确输出JSON格式要求 | 引导LLM朝正确方向生成 |
| **② Schema层** | Pydantic model_validate_json() | 强制校验字段完整性、类型、范围 |
| **③ 重试层** | LLM_MAX_RETRIES=1 | 解析失败后重试1次（共2次机会）|
| **④ 兜底层** | 返回默认模板行程 | 重试仍失败时的最后保障 |

**重试流程**：
```python
for attempt in range(LLM_MAX_RETRIES + 1):  # 0, 1 (共2次)
    try:
        response = llm.invoke(prompt)
        parsed = Itinerary.model_validate_json(response.content)
        return parsed
    except ValidationError as e:
        logger.warning(f"LLM输出解析失败(第{attempt+1}次): {e}")
        if attempt == LLM_MAX_RETRIES:
            return get_default_itinerary(trip_request)  # 兜底
        continue
```

**日期解析的兼容性处理**（刚刚修复的Bug）：

除了常规Pydantic校验外，还增加了`@field_validator`来兼容前端传入的各种日期格式：

```python
@field_validator("start_date", "end_date", mode="before")
def _parse_flexible_date(cls, v):
    """兼容 '2026-6-6' / '2026/8/15' 等非零填充格式"""
    # 尝试标准ISO格式
    for fmt in ("%Y-%m-%d", "%Y/%m/%d", "%Y.%m.%d"):
        try: return datetime.strptime(v, fmt).date()
        except ValueError: continue
    # 兼容非零填充：正则匹配 YYYY-M-D
    m = re.match(r"^(\d{4})[-/.](\d{1,2})[-/.](\d{1,2})$", v)
    if m:
        return DateType(int(m.group(1)), int(m.group(2)), int(m.group(3)))
    raise ValueError(f"无法解析日期 '{v}'")
```

### 4.2 Prompt Engineering

### Q31: 你在Prompt设计方面有什么经验和技巧？

#### 标准答案

**本项目中的三类Prompt设计模式**：

**(1) Query Rewrite Prompt — 角色扮演 + 格式约束**

```python
"你是一个 RAG 检索 query 改写专家。"        # 明确角色
"只输出检索关键词，用空格分隔"                # 正面格式约束
"不要输出解释、标点或任何多余文字"            # 负面约束
"关键词要具体，优先包含景点名称"              # 质量引导
```

**技巧**：角色定义越具体，LLM的表现越好。"专家"比"助手"更能产生专业输出。

**(2) 行程生成 Prompt — 结构化示例 + Few-Shot**

行程生成的Prompt通常包含：
- **角色定义**："你是一位资深旅行规划师"
- **输入格式说明**：明确destination/days/budget/preferences的含义
- **输出格式要求**：严格的JSON Schema描述
- **Few-Shot示例**：给出1-2个高质量的输入→输出范例
- **约束条件**：预算限制、每天最多4个景点、避免行程过于紧凑

**(3) Rerank Instruct Prompt — 选择偏好引导**

```python
"给定一个旅行规划查询，从候选文档中检索出最具体、最详细的片段。"
"优先选择包含具体景点名称、活动推荐、实用信息的片段，"
"避免选择泛化的目的地简介、文档开头等信息量低的片段。"
```

**技巧**：告诉模型"不要什么"和"要什么"同等重要。避免模型选择那些看起来信息量大但实际上是废话的长文本。

**通用Prompt设计原则**：

| 原则 | 说明 | 示例 |
|------|------|------|
| **角色具体化** | 不要说"你是助手"，要说"你是XX领域的专家" | "RAG检索query改写专家" |
| **格式显式化** | 明确指定输出格式，不要让LLM猜 | "只输出关键词，空格分隔" |
| **负面约束** | 明确说不要什么 | "不要输出解释、标点" |
| **分解复杂任务** | 一个Prompt只做一件事 | Query Rewrite和行程生成分成两个独立调用 |
| **温度参数化** | 根据任务类型选择合适温度 | 创意0.7 / 翻译0.3 / 提取0.1 |

```python
class TraceRecord(BaseSchema):

    cache_hit: bool                   # 是否命中缓存
    
    # LLM调用层
    llm_calls: list[LLMCallRecord]    # 每次LLM调用的详细信息

    agent_steps: list[AgentStepRecord] # Agent每步的thought/action/observation
    agent_type: str                   # Agent类型(ReAct/MultiStage)
    
    # 输出层
    final_output: dict                # 最终输出
    success: bool                     # 是否成功
    error_message: str                # 错误信息
    
    # 统计
    total_duration_ms: int            # 总耗时
    total_tokens_used: int            # 总token消耗
    total_cost: float                 # 估算成本
```

**追踪的完整链路**：

```
用户请求进入
  ↓

  ↓
TripRequest 校验解析            ← 记录 user_request, destination
  ↓
RAG 检索                        ← 记录 rag_contexts, rag_duration_ms, cache_hit
  ↓
LLM 调用 (qwen-max)            ← 记录 model, prompt, response, tokens, cost, duration
  ↓
Agent 执行 (ReAct)              ← 记录每步 state/thought/action/observation/duration
  ↓
多阶段流水线                     ← 记录每个 stage 的 status/duration/warnings
  ↓
返回结果                        ← 记录 final_output, success, total_duration
  ↓
TraceContext 持久化到 SQLite     ← 可通过 /traces API 查询
```

**数据校验策略（Pydantic v2）**：

```python
class BaseSchema(BaseModel):
    model_config = {"extra": "ignore"}  # 忽略未知字段，兼容前端多发字段

class TripRequest(BaseSchema):
    destination: str = Field(..., description="目的地")
    start_date: DateType = Field(...)    # 带 @field_validator 兼容多种格式
    end_date: DateType = Field(...)
    travelers: int = Field(..., ge=1)    # ge=1: 最少1人
    days: int = Field(default=3, ge=1, le=14)  # 1-14天
    budget: float = Field(..., ge=0)    # ≥0
```

**校验层次**：

| 层级 | 手段 | 位置 |
|------|------|------|
| **类型校验** | Pydantic Field类型注解 | Schema定义 |
| **范围校验** | `ge`/`le`/`gt`/`lt` 约束 | Field参数 |
| **格式校验** | `@field_validator` 自定义校验 | 如日期格式兼容 |
| **业务校验** | Route handler中的逻辑判断 | 如`end_date >= start_date` |
| **全局异常处理** | Starlette exception middleware | 统一返回4xx/5xx错误 |

**错误响应格式**：
```json
{
  "detail": "Failed to parse TripRequest: 2 validation errors for TripRequest..."
}
```

### 5.5 性能优化经验

### Q37: 你做过哪些性能优化？效果如何？

#### 标准答案

**优化清单及效果**：

| # | 优化点 | 方法 | 效果 |
|---|--------|------|------|
| 1 | **Rerank缓存** | 对相同query+相同候选集的Rerank结果缓存到Redis | 命中时延迟 **728ms→425ms (-41.6%)** |
| 2 | **RAG缓存** | 混合检索结果缓存到Redis（TTL=6h）| 重复查询延迟 **500ms→10ms** |
| 3 | **降本优化** | 候选chunk数 ≤ top_k时跳过LLM Rerank | ~10%请求省掉Rerank API调用 |
| 4 | **噪声预过滤** | Rerank前过滤已知噪声chunk | 减少无效API调用 + 提升精度 |
| 5 | **批量Embedding** | `chunk_size=10` 批量编码 | 减少网络往返次数 |
| 6 | **内存LRU缓存** | 热点数据常驻进程内存 | L1命中延迟 <1ms |
| 7 | **惰性导入** | redis/chromadb/httpx等重型依赖用时再加载 | 启动速度提升50%+ |
| 8 | **MCP工具内置缓存** | 天气/地图API结果各自缓存 | 避免重复调用外部服务 |

**性能基线数据（参考值）**：

| 场景 | P50延迟 | P95延迟 | P99延迟 |
|------|---------|---------|---------|
| FAST模式（仅行程规划）| ~2s | ~3s | ~5s |
| FULL模式（全部5阶段）| ~8s | ~12s | ~15s |
| 聊天意图识别+路由 | ~1s | ~2s | ~3s |
| RAG检索（缓存未命中）| ~500ms | ~800ms | ~1500ms |
| RAG检索（缓存命中）| ~10ms | ~20ms | ~50s |

## 七、场景题与开放性问题

### Q39: 如果用户量增长10倍，你的系统会遇到什么瓶颈？怎么解决？

#### 标准答案

**瓶颈分析和解决方案**：

| 瓶颈 | 当前状态 | 10倍后预期 | 解决方案 |
|------|---------|-----------|---------|
| **Python GIL锁** | 单进程FastAPI | CPU成为瓶颈 | ① Uvicorn多Worker(gunicorn) ② 异步化(async/await) ③ 热点路径用Go/Rust重写 |
| **LLM API限流** | DashScope有QPS限制 | 请求被拒 | ① 请求队列 + 令牌桶限流 ② 多Key轮询 ③ 本地小模型分流简单请求 |
| **ChromaDB并发** | 嵌入式，写锁互斥 | 写入阻塞 | ① 迁移到Milvus/Qdrant ② 读写分离（主从ChromaDB）|
| **SQLite并发** | 单写者多读者 | 写入排队 | ① 迁移到PostgreSQL（已有配置支持）② 连接池扩大 |
| **Redis单点** | 单实例 | 内存/连接数不足 | ① Redis Cluster ② 增加Sentinel哨兵 |
| **内存缓存失效** | 进程内缓存不共享 | 多Worker时缓存利用率低 | ① 全部走Redis ② 引入分布式缓存(如GroupCache) |

**优先级排序**：
1. **P0（立即）**：PostgreSQL替代SQLite + Uvicorn多Worker部署
2. **P1（近期）**：ChromaDB → Milvus迁移 + Redis Cluster
3. **P2（中期）**：LLM调用异步化 + 请求队列
4. **P3（长期）**：热点路径性能优化（Rust扩展或Go微服务）

### Q40: 如果DashScope API全面不可用（如运营商故障），你的系统还能工作吗？降级链条是怎样的？

#### 标准答案

**降级链条（由浅入深）**：

```
Level 0: 完整模式（所有服务正常）
  ├─ LLM: qwen-max (DashScope)
  ├─ Embedding: text-embedding-v4 (DashScope)
  ├─ Rerank: qwen3-rerank (DashScope)
  ├─ 地图: 高德REST API
  └─ 天气: 高德天气API

Level 1: DashScope LLM不可用
  └─ 行程生成功能完全不可用（目前没有本地模型Fallback）
     → 返回友好错误提示："AI服务暂时不可用，请稍后重试"

Level 2: DashScope Embedding不可用
  ├─ RAG向量检索不可用
  ├─ Fallback到BM25关键词检索（纯规则，不依赖Embedding）
  └─ Rerank Fallback到规则级打分（不依赖Cross-encoder）
     → 质量下降但系统可用（Top1: 93.3% → ~75%）

Level 3: DashScope Rerank不可用
  ├─ Rerank自动Fallback到规则级打分
  └─ 质量轻微下降（Top1: 93.3% → ~85%）

Level 4: 高德API不可用
  ├─ 地图Stage自动SKIPPED
  ├─ 天气Stage自动SKIPPED
  └─ 行程仍然生成，只是缺少POI坐标和天气信息

Level 5: Redis不可用
  ├─ 自动降级到内存LRU缓存
  └─ 缓存无法跨进程共享（多Worker模式下）

Level 6: 全部外部服务不可用
  └─ 仅剩：规则级Query Rewrite + BM25检索 + 规则级Rerank + SQLite存储
     → 能生成基础行程，但质量和丰富度大幅下降
```

**改进方向**：可以在Level 1增加本地模型Fallback（如Ollama + Qwen2.5-7B），实现完全离线运行。

### Q41: 你的项目还有什么可以改进的地方？

#### 标准答案

**短期改进（1-2周）**：
1. **TTL随机化**：在缓存TTL基础上加减随机偏移量，防止缓存雪崩
2. **请求限流**：添加Rate Limiter中间件，防止单用户过量请求
3. **输入消毒**：对用户输入进行XSS/注入防护（虽然Pydantic已做类型校验）
4. **健康检查接口**：添加`/health`端点，返回各依赖服务的连通性状态

**中期改进（1-2月）**：
5. **异步化改造**：将多阶段流水线的独立Stage改为asyncio并行执行
6. **向量化评估自动化**：将评估脚本集成到CI/CD，每次PR自动运行回归测试
7. **用户反馈闭环**：在行程结果页添加" thumbs up/down "按钮，收集用户满意度数据
8. **知识库增量更新**：支持在不重建全量索引的情况下增删改单个文档

**长期改进（3-6月）**：
9. **多模态RAG**：将攻略图片纳入检索范围（CLIP图像编码）
10. **对话式行程迭代**：基于多轮对话逐步完善行程（而非一次性生成）
11. **A/B测试框架**：在线验证不同RAG策略/LLM Prompt的效果差异
12. **国际化(i18n)**：支持英文等多语言界面和行程生成
